# 괴이 기록국 프로그래밍 기획서 · 로드맵/MVP

> 문서 위치: `[기획서]/02_프로그래밍기획서_로드맵_MVP.md` | 상태 기준: `codex/mvp048-campaign-novel`의 `4cf39a2` (진행 브랜치) | 상세 부록: [`MVP_ROADMAP.md`](../MVP_ROADMAP.md), [`docs/planning/MVP044_046_EXECUTION_PLAN.md`](../docs/planning/MVP044_046_EXECUTION_PLAN.md)

## 1. 목적과 기술 기준

이 문서는 프로그래밍 팀의 활성 원본이다. 어떤 기능을 어느 상태 소유자·데이터·화면에 연결하고, 어떤 저장 위험과 검증 게이트를 통과해야 하는지 정한다.

| 항목 | 기준 |
|---|---|
| 엔진 | Godot 4.7 stable / GDScript |
| 플랫폼 | PC, 16:9, 마우스·키보드, 1280×720·1920×1080 |
| 상태 소유 | `scripts/core/game_state.gd`, `scripts/core/campaign_state.gd` |
| 콘텐츠 | `data/episodes/`, `data/daily_episodes.json`, 카탈로그 스크립트 |
| 화면 | `scenes/`, `scripts/scenes/`, 상태를 소유하지 않는 `scripts/ui/` |
| 저장 원칙 | 기존 ID와 기록을 보존하고, 신규 필드·버전은 이관과 저장 왕복 테스트가 있을 때만 추가 |

보호 경로는 `scripts/core/game_state.gd`, `data/episodes/*`, `project.godot`, `knowledge/base-pack/*`다. 이 경로는 Codex가 직접 변경·검증한다. 비보호 초안이나 읽기 감사만 위임할 수 있으며, 실제 저장·게임 규칙·최종 통합 책임은 위임하지 않는다.

## 2. 구현 구조와 소유권

| 층 | 책임 | 해서는 안 되는 일 |
|---|---|---|
| GameState/CampaignState | 저장, 진행, 보상 멱등성, 사건·계약·관계 기록 | UI 배치·연출 상태 소유 |
| 사건 JSON·카탈로그 | 지점, 선택, 조건, 문구, 등록 가능한 콘텐츠 | 저장 이관을 독자적으로 수행 |
| 씬 스크립트 | 입력, 화면 흐름, 상태 API 호출 | 상태를 중복 저장 |
| UI 컴포넌트 | 표시와 선택 ID 신호 | 정답·보상·진행을 직접 결정 |
| 테스트 | 계약·회귀·저장 왕복 증명 | 사용자 저장이나 캡처 원본 변경 |

미니게임은 진행 중 보드를 저장하지 않는다. 3×3 학습은 완료·결과 저장을 발생시키지 않고 4×4 최종 검증만 `completed`와 결과 저장을 한 번 발생시킨다. 연출·표정·컷인은 기존 상태를 표현할 뿐 진행을 소유하지 않는다.

## 3. MVP 이력과 현재 구현선

| MVP | 범위 | 브랜치 기준 상태 | 다음 게이트 |
|---|---|---|---|
| 043 | 저승역 조사/3×3·4×4/회수, 5인, 아카, 전용 UI | 구현 확인 | 실제 화면 회귀를 유지 |
| 044 | AFTER·DAILY·FACTION 선택형 서사 | 진행 브랜치에 구현 단위 존재 | 재진입·DB·실제 플레이 검증 |
| 045 | 관계 기록·선택 기억·연속 이벤트 | 진행 브랜치에 구현 단위 존재 | 12체인/30장면의 해금·후속 반응 감사 |
| 046 | 표정·컷인·접근성 폴백 | 진행 브랜치에 표현 계층 존재 | 해상도·입력·폴백 실제 캡처 |
| 047 | 연구·잔향 파편·세력/계약 기반 | 진행 브랜치에 구현 단위 존재 | 경제가 추리를 자동 해결하지 않는 회귀 |
| 048 | 네 번째 사건, 추가 연구·외부 계약, 텍스트 노벨 셸 | 진행 중 | 사건 완주, 계약 멱등성, 보고서/DB, 수동 UI QA |

이 표는 `main` 승격을 뜻하지 않는다. 현재 브랜치와 커밋의 실제 검증 결과만 구현 확인으로 올린다.

## 4. 다음 구현 순서

1. **MVP-048 닫기:** 네 번째 사건의 시작·조사·위험 사례·안정화·결과·DB를 저장 왕복 포함으로 완주한다. 기존 세 사건과 Day 8/10을 회귀한다.
2. **추리 계약 통일:** 주요 판단의 후보 수, 위험 후 복귀, 사례 기반 비활성화, 아카 안내가 사건마다 같은 페어플레이 원칙을 지키는지 점검한다.
3. **표현·접근성 QA:** 텍스트 노벨 장면·저승역 전용 UI·설정 화면을 1280×720/1920×1080에서 캡처한다.
4. **승격 판단:** 전체 회귀, 문서 상태, 사용자 변경 보존을 확인한 뒤에만 통합 브랜치를 `main`으로 승격한다.

네 번째 사건 이후의 새 경제, 새 사건, 용병 고용 확장은 위 항목을 닫은 뒤 별도 MVP로 승인한다.

## 5. 저장·호환 계약

- 기존 사건 ID, 단서, 미니게임, 보고서, 일상 완료 기록은 유지한다.
- 이벤트·계약·보상은 기록 ID로 멱등 처리한다. 재진입, 재시작, 재로드가 비용·효과·보상을 중복 적용해서는 안 된다.
- 계약은 사건당 최대 하나다. 예약은 무료, 사건 시작 시에만 파편을 차감하고 취소는 무료다.
- 새 저장 버전은 누락된 새 사건의 기본 상태만 보완하며 기존 진행, 일정, 단서, 결과를 초기화하지 않는다.
- 사용자 소유 `.import`, 생성 `.uid`, `project.godot`, 수동 캡처는 출처와 승인 없이 수정·스테이징·커밋하지 않는다.

## 6. 커밋·검증 게이트

각 목적 단위는 기능 하나와 그 계약 테스트를 함께 커밋한다. 커밋 전 최소 검증은 다음과 같다.

```text
JSON 파싱 또는 데이터 로드
→ 관련 Godot headless 계약 테스트
→ 저장 왕복(상태·경제·계약·결과를 바꿀 때)
→ git diff --check
→ 영향 화면의 1280×720·1920×1080 수동 확인
```

모든 테스트는 별도 임시 `APPDATA`에서 순차 실행해 사용자 저장과 병렬 충돌을 막는다. 실패한 테스트, 미확인 실제 화면, 보호 경로 위험은 완료로 쓰지 않는다. 전체 회귀와 문서 동기화 전에는 `main`에 승격하지 않는다.

### 작업 시작 체크

- [ ] 플레이어 가치, 포함/제외 범위, 영향 파일, 저장/UI 위험, 검증 방법을 적었다.
- [ ] 다섯 활성 기획서 중 갱신 대상과 `변경 없음` 대상을 표시했다.
- [ ] 최신 이미지가 필요한 작업이면 아트 기획서와 이미지 인덱스를 읽었다.

### 커밋 전 체크

- [ ] 바뀐 규칙·시나리오·UX·밸런스는 게임 기획서에 반영했다.
- [ ] 바뀐 데이터·저장·MVP 상태·테스트 게이트는 이 문서에 반영했다.
- [ ] 바뀐 자산·UI 방향·스크린샷은 아트 기획서와 이미지 인덱스에 반영했다.
- [ ] 바뀐 BGM/SFX/보이스·무음 대체는 사운드 기획서에 반영했다.
- [ ] 실행한 검증과 미검증은 QA 기획서에 반영했다.
- [ ] 세부 명세가 중복되면 이 다섯 문서에는 링크만 두고 부록으로 내렸다.

기존 승인 문서·Goal·감사·테스트는 수치와 계약을 증명하는 **부록**이다. 고유 계약이 사라진 문서만 참조 검사 후 `docs/archive/backup/YYYY-MM-DD/`에 원래 경로, 사유, 대체 문서를 적어 보관한다.
