# Godot 네이티브 UI 구조 적용 원칙

> 현행 기준: `MVP-043 + CORE-VALIDATION-001` / `Ver 4.2` / 저장 스키마 `mvp-039`


## 검토 결론

GPT의 Godot 네이티브 UI Toolkit 제안은 방향을 채택하되 현재 프로젝트 구조에 맞게 축소 적용한다. `Control`, `Container`, `Theme`, 재사용 Scene, 명시적 Signal은 사용하지만 별도 UI 프레임워크를 만들지 않는다.

프로젝트에는 이미 `UiThemeFactory`, UI 자산 카탈로그, F2 런타임 편집기와 `UiLayoutStore`가 있다. 따라서 새 테마 기반, EditorPlugin, 두 번째 레이아웃 저장 계층을 만드는 대신 기존 시스템을 확장한다.

## 프로젝트 적용 원칙

- 화면 배치는 Godot `Container`가 담당하고 반복되는 표시·입력 단위만 Scene으로 분리한다.
- 재사용 컴포넌트는 화면 상태를 직접 소유하거나 `GameState`를 수정하지 않는다.
- 컴포넌트는 부모가 계산한 표시 데이터를 받고 사용자 의도를 Signal로 반환한다.
- 색상과 반복 상태 스타일은 `UiThemeFactory`에 두고 화면별 임의 스타일 복제를 피한다.
- 현재 `scripts/ui/`, `scripts/scenes/`, `scenes/` 구조를 유지하며 새 최상위 `ui/` 계층을 만들지 않는다.
- 동적 목록의 개별 항목은 F2 런타임 편집 대상으로 등록하지 않는다. 화면의 안정된 상위 영역만 기존 편집 정책을 따른다.
- PC 16:9와 마우스·키보드를 기본으로 하며 모바일·게임패드는 명시된 작업에서만 추가한다.

## 첫 파일럿: 준비 화면 요원 카드

`AgentSelectionCard`는 요원 이름·성향·직책·역할, 체력·정신력, 5개 능력치, 설명과 선택 상태를 표시한다. 입력은 다음 두 신호로만 전달한다.

- `selection_requested(agent_id)`
- `detail_requested(agent_id)`

준비 화면은 기존대로 `GameState` 조회, 선택·해제, 최대 인원 판단, 상세 정보 구성과 임무 시작 상태를 소유한다. 상세 패널은 한 화면에서만 사용하므로 별도 범용 컴포넌트로 추출하지 않는다.

선택 카드는 기존 테마의 `AgentCard`, `AgentCardSelected` 변형을 사용한다. 긴 한국어 이름과 설명에는 스마트 단어 줄바꿈을 적용하고 능력치는 폭이 부족하면 다음 줄로 흐르게 한다.

## 제외 범위

- Unity UI Toolkit 모사, 마크업·CSS, 가상 DOM, 범용 데이터 바인딩
- EditorPlugin 또는 새 런타임 UI 편집기
- 준비 화면 전체 재설계와 다른 화면의 일괄 변환
- 요원 데이터, 선택 규칙, 저장 스키마와 화면 전환 변경
- 신규 대사, 이미지 자산과 게임 시스템

## 검증과 후속 판단

컴포넌트 계약, 준비 화면 연결, Godot 전체 파싱과 기존 요원·저장·런타임 편집기·로그 테스트를 통과한 뒤 다른 반복 UI에 같은 원칙을 적용할지 판단한다. 해상도 확인은 1280×720과 1920×1080을 기준으로 한다.

후속 전체 캠페인 회귀에서 사용자 승인에 따라 `agent_oh_hyun`의 데이터 성향 `balanced`를 `GameState.ALLOWED_AGENT_TEMPERAMENTS`에 허용했다. 저장 스키마와 에피소드 데이터는 바꾸지 않았으며, 현행 검증은 `tests/run_godot_regression.sh`와 `TEST_CHECKLIST.md`를 기준으로 3인 편성·저장 재개·회수 보조를 확인한다.

## Base 승격 후보

“엔진 네이티브 UI를 도입할 때 새 프레임워크보다 프로젝트의 기존 테마·편집·레이아웃 체계를 먼저 조사하고, 상태를 소유하지 않는 작은 파일럿 컴포넌트로 검증한다”는 원칙은 Base 승격 후보다. 이번 작업에서는 Base 원격을 변경하지 않는다.
