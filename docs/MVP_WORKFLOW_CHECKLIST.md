# MVP Workflow Checklist

## 멀티모델 위임 추가 확인

- [ ] `route` 결과의 provider, 권한, 산출물 계약, Codex 보호 여부를 확인했다.
- [ ] `prepare` 묶음의 입력 경로·크기·SHA-256이 실제 파일과 일치한다.
- [ ] 외부 patch/review 또는 이미지 manifest가 모두 도착했고 경로 이탈·보호 경로·정보 누설이 없다.
- [ ] DeepSeek 중단/일시 오류 재개는 동일 세션 한 차례뿐이며 결과 회수 여부가 기록되었다.
- [ ] 실제 diff와 실행 테스트는 Codex가 독립 확인했다.
- [ ] 20건 누적 시 직접 작성 비율·첫 통과율·회수율·재작업률을 재평가한다.

대사·튜토리얼·상황별 반응문이 포함된 작업은 `docs/DIALOGUE_AUTHORING_WORKFLOW.md`를 추가로 적용한다. 외부 GPT의 `dialogue_rewrite.patch`와 `dialogue_review.md`, DeepSeek 읽기 전용 검수, Codex의 실제 diff·JSON·Godot 검증이 모두 있어야 적용 완료로 판단한다.

> **Base 기반 로컬 사본 + 도시괴담 기록국 확장**: 공용 MVP 체크리스트의 원본과 기준 커밋은 `docs/BASE_RULES_VERSION.md`를 따른다. 이 문서는 현재 프로젝트의 역할과 Godot 검수 흐름을 추가로 기록한다.

이 문서는 `도시괴담 기록국`의 실제 MVP 작업 순서와 시작/완료 체크리스트다. 엔진과 장르에 종속되지 않는 원칙은 `docs/AI_SHARED_WORK_RULES.md`, Godot와 기록국의 구현 규칙은 `AGENTS.md`와 `docs/CODEX_SHARED_WORK_RULES.md`에서 확인한다.

## 1. 역할

- ChatGPT/기획 AI: 의도 분석, 플레이어 경험, 필요한 근거 조사, 시스템 설계, Issue, Goal, 검수 기준을 담당한다. 직접 코딩하지 않는다.
- Codex/구현 AI: Issue와 Goal을 기준으로 실제 GDScript, JSON, TSCN, README를 수정하고 검증한다.
- 사용자: 방향 결정, Godot 실행 확인, 플레이 감각 확인, 최종 승인과 범위 판단을 담당한다.

## 2. 표준 순서

```text
0. 작업 유형 분류
1. 최신 사용자 지시 확인
2. 공용 규칙과 프로젝트 규칙 확인
3. Issue / 이전·다음 MVP / 현재 파일 기준선 확인
4. Definition of Ready 확인
5. 필요한 경우 근거 조사와 벤치마킹
6. 범위·제외·위험·영향 파일 확정
7. GitHub Issue 작성 또는 수정
8. 필요한 Goal / 문서 갱신
9. 구현 전 파일·계획 보고
10. Codex 구현
11. 자동·정적 검증
12. Godot 수동 검수
13. 결과 보고와 짧은 회고
14. 반복 규칙만 공용 문서에 승격
```

문서/오탈자 작업은 0 → 1 → 3 → 9 → 11 → 13의 빠른 경로를 사용할 수 있다. 이미 확정된 단순 구현은 불필요한 벤치마킹을 반복하지 않는다.

## 3. 작업 시작 체크리스트

```md
- [ ] 최신 사용자 지시와 요청 권한을 확인했다.
- [ ] 기획, 문서, 구현, 버그, 검수 중 작업 유형을 분류했다.
- [ ] 현재 브랜치와 작업 상태를 확인했다.
- [ ] AGENTS.md 또는 CODEX_RULES.md를 확인했다.
- [ ] 공용 규칙과 기록국 전용 규칙을 확인했다.
- [ ] 현재 Issue와 이전/다음 MVP 연결을 확인했다.
- [ ] 실제 수정 대상 파일과 기존 사용자 변경사항을 확인했다.
- [ ] 목표가 한 문장으로 설명된다.
- [ ] 구현 범위와 제외 범위가 분리되어 있다.
- [ ] 완료 기준이 조건·행동·관찰 결과로 작성되어 있다.
- [ ] 저장/불러오기, 씬 이동, 데이터 호환성 위험을 적었다.
- [ ] Godot에서 확인할 화면 순서가 있다.
```

## 4. 조사와 벤치마킹 체크리스트

새 시스템, 핵심 루프, UI/UX 방향, 범위 변경에서만 수행한다.

```md
- [ ] 관련 사례 3~5개를 우선 선정했다.
- [ ] 장르 핵심 구조나 큰 방향 전환일 때만 10개 이상으로 확장했다.
- [ ] 조사 시간과 질문을 먼저 제한했다.
- [ ] 사례별 관찰점, 실제 반응, 기록국 적용 이유를 구분했다.
- [ ] 공식 자료/개발자 발언/사용자 반응을 섞어 단정하지 않았다.
- [ ] 반드시 반영, 조건부 반영, 제외 결론을 작성했다.
- [ ] 출처 URL과 확인 날짜를 남겼다.
```

상세 형식은 `docs/BENCHMARKING_REFERENCE_GUIDE.md`를 따른다.

## 5. Issue Definition of Ready

```md
- [ ] 목표와 해결할 플레이어 문제가 있다.
- [ ] 핵심 흐름과 사용자 가치가 있다.
- [ ] 이전 MVP에서 받는 상태와 다음 MVP로 넘기는 상태가 있다.
- [ ] 구현 범위와 제외 범위가 작게 잘렸다.
- [ ] 데이터, 저장값, 씬/노드 영향이 적혀 있다.
- [ ] 완료 기준마다 검증 방법이 연결된다.
- [ ] 예상 위험과 대응이 있다.
- [ ] 보고 형식과 사용자의 Godot 확인 항목이 있다.
```

준비 기준을 만족하지 못하면 Codex Goal을 만들지 않고 Issue를 보완한다.

## 6. HTML 대시보드 체크리스트

MVP-028부터 HTML 대시보드는 기본 산출물에서 제외한다. 사용자가 명시적으로 요청하면 별도 작업 범위와 검증 기준을 먼저 정한다.

## 7. Codex Goal 체크리스트

```md
- [ ] Issue 번호와 목표가 맞다.
- [ ] 공용 규칙과 `docs/CODEX_SHARED_WORK_RULES.md`를 참조한다.
- [ ] 벤치마킹은 필요할 때 문서 참조 한 줄로만 둔다.
- [ ] 실제 구현 범위와 제외 범위가 분명하다.
- [ ] 완료 기준이 테스트 가능하다.
- [ ] 요청하지 않은 HTML과 장기 기획이 섞이지 않았다.
- [ ] 작업 전 확인 파일이 있다.
- [ ] 작업 후 보고 형식이 있다.
- [ ] Goal 뒤에 구현 지시와 무관한 문장을 붙이지 않았다.
```

## 8. 구현 전 보고

구현 AI는 수정 전에 다음만 짧게 보고한다.

```md
수정 대상 파일:
-

변경 계획:
1.
2.

제외 범위:
-
```

그 뒤 실제 파일과 참조 위치를 확인하고, 요청 범위 밖 리팩터링은 하지 않는다.

## 9. Definition of Done

```md
- [ ] 완료 기준을 실제 파일과 실행 결과로 확인했다.
- [ ] 기존 MVP 흐름과 사용자 변경사항을 보존했다.
- [ ] 저장/불러오기와 데이터 호환성을 확인했다.
- [ ] 변경된 씬, 데이터, 핵심 호출 경로를 검증했다.
- [ ] 자동/정적 검증 결과가 있다.
- [ ] Godot에서 사용자가 따라 할 화면 순서가 있다.
- [ ] 검증하지 못한 항목을 명시했다.
- [ ] 남은 위험 중 현재 줄일 수 있는 것은 처리했다.
- [ ] diff와 변경 파일이 요청 범위 안이다.
- [ ] README 또는 관련 규칙 문서가 필요한 만큼 갱신됐다.
- [ ] Serena 사용 여부를 사실대로 보고한다.
- [ ] 서사형 안내자의 역할이 담당 시스템과 일치하고 최초/반복 안내가 분리되어 있다.
- [ ] 음향 안내와 동등한 시각·문구 안내가 있으며 미획득 정보나 정답을 누설하지 않는다.
```

DoD는 모든 프로젝트에 같은 테스트를 강제하는 목록이 아니라, 해당 작업의 완료를 판단할 수 있는 최소 공통 기준이다.

### MVP-038 추가 확인

- [ ] 현재 반일 결과를 확인하기 전에 다음 반일을 편성할 수 없다.

## 11. 다섯 MVP마다 회고

MVP-005, MVP-010, MVP-015처럼 5개 단위로 다음만 점검한다.

```md
- [ ] 실제로 사용하지 않는 규칙을 삭제하거나 줄였다.
- [ ] 반복된 오류의 원인과 재발 방지를 기록했다.
- [ ] 공용 규칙으로 승격할 내용과 기록국 전용 내용을 분리했다.
- [ ] 검증 명령과 Godot 수동 검수가 현재 구조와 맞다.
- [ ] 다음 5개 MVP에 필요한 문서만 남겼다.
```
