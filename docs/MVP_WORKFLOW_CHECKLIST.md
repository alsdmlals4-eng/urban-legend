# MVP Workflow Checklist

이 문서는 `도시괴담 기록국` 프로젝트에서 ChatGPT, Codex, 새 AI, 사용자 모두가 같은 순서로 작업을 이어가기 위한 실제 작업 체크리스트다.

`AGENTS.md`, `docs/AI_WORKFLOW_RULES.md`, `docs/CODEX_SHARED_WORK_RULES.md`, `docs/BENCHMARKING_REFERENCE_GUIDE.md`와 함께 확인한다.

---

## 1. 역할 고정

### ChatGPT 역할

ChatGPT는 직접 코딩하지 않는다.

담당:

- 기획 의도 파악
- 플레이어 경험 분석
- 벤치마킹 분석
- 시스템 설계
- 데이터 구조 설계
- HTML 대시보드 설계/갱신
- GitHub 문서 정리
- GitHub Issue 작성/수정
- Codex Goal 작성
- Codex 작업 검토
- 테스트 체크리스트 작성
- 결과 회고와 다음 작업 정리

### Codex 역할

Codex는 실제 코딩 담당자다.

담당:

- GitHub Issue와 Goal 기준 실제 코드 작성
- Godot GDScript 수정
- JSON 데이터 수정
- TSCN 씬 수정
- README.md 갱신
- 테스트 실행 또는 정적 확인
- 변경 파일/구현 방식/검증 결과/남은 위험 보고

### 사용자 역할

사용자는 방향을 결정하고 결과를 검수한다.

담당:

- 기획 방향 결정
- HTML/시트 확인 및 수정
- GitHub Issue/문서 확인
- Godot 실행 테스트
- 오류 메시지 제공
- 최종 승인

## 2. 표준 작업 순서

```text
0. 작업 유형 분류
1. 사용자 최신 지시 확인
2. GitHub 기준 문서 확인
3. 현재 MVP / Issue 기준선 확인
4. 작업 준비 상태 체크
5. 벤치마킹 필요 여부 판단
6. 벤치마킹 분석
7. 구현 범위 / 제외 범위 확정
8. 위험 분석 / 영향 파일 예상
9. GitHub Issue 작성 또는 수정
10. HTML 전체 흐름 대시보드 갱신
11. Codex Goal 작성
12. Codex 구현
13. Godot 검수
14. 결과 보고 / 회고
15. 필요한 경우 GitHub 공동 규칙 갱신
```

## 4. 좋은 프롬프트 변환 체크리스트

사용자의 짧은 요청을 그대로 Codex에게 넘기지 않는다.

```md
- [ ] 목표
- [ ] 배경
- [ ] 사용자의 의도
- [ ] 플레이어 경험
- [ ] 구현 범위
- [ ] 제외 범위
- [ ] 데이터 요구사항
- [ ] Godot 씬/노드 영향
- [ ] 완료 기준
- [ ] 테스트 방법
- [ ] 결과물 형식
```

흐름:

```text
사용자 요청
→ 의도 파악
→ 부족한 조건 확인
→ 좋은 프롬프트로 재작성
→ 기획 / Issue / Goal / 테스트 기준으로 변환
→ 사용자 확인 또는 바로 진행
```

## 6. 벤치마킹 체크리스트

벤치마킹이 필요한 작업:

- 새 시스템 추가
- MVP 방향 변경
- UI/UX 변경
- 게임 루프 변경
- 요원/장비/기록물/회수전 같은 핵심 구조 변경
- 새 프로젝트 시작

표 형식은 `docs/BENCHMARKING_REFERENCE_GUIDE.md`를 따른다.

```md
| # | 게임/사례 | 관찰포인트 | 반응 | 기록국 적용(+사유) |
|---:|---|---|---|---|
```

체크:

```md
- [ ] 유사/인접 사례 10개 이상을 확인했는가?
- [ ] 게임/사례를 적었는가?
- [ ] 관찰포인트를 적었는가?
- [ ] 반응을 적었는가?
- [ ] 기록국 적용(+사유)을 적었는가?
- [ ] 반영할 것과 제외할 것을 나눴는가?
- [ ] 출처 없는 일반론으로 판단하지 않았는가?
```

## 8. HTML 대시보드 체크리스트

HTML 대시보드는 사람과 AI가 함께 보는 프로젝트 지도다.

포함할 것:

```md
- [ ] 프로젝트 개요
- [ ] 핵심 루프 다이어그램
- [ ] MVP 진행 보드
- [ ] 기능 관계도
- [ ] 조사/대화/회수/결과 흐름
- [ ] 요원/장비/기록물 관계
- [ ] UI 와이어프레임 또는 카드 예시
- [ ] 기획 누락 체크리스트
- [ ] 필요한 경우 벤치마킹 3~5개 요약
- [ ] 다음 작업 후보
```

작성 원칙:

```md
- [ ] 한 파일로 열 수 있게 만든다.
- [ ] 복잡한 빌드 도구를 쓰지 않는다.
- [ ] HTML + CSS + 간단한 JS만 사용한다.
- [ ] 텍스트는 사용자가 바로 수정할 수 있게 둔다.
- [ ] 표와 카드 구조를 활용한다.
- [ ] 흐름은 화살표와 카드로 표시한다.
- [ ] PC에서 보기 좋은 대시보드를 우선한다.
- [ ] data-testid를 주요 탭/카드/섹션에 넣는다.
```

HTML → GitHub 반영 흐름:

```text
HTML 대시보드에서 기획 확인
→ 사용자와 ChatGPT가 내용 수정
→ 방향성 확정
→ GitHub PROJECT_BRIEF.md / DESIGN_INTENT.md 갱신
→ GitHub Issue 생성
→ Codex Goal 작성
→ Codex 구현
→ Godot 테스트
→ HTML 대시보드와 GitHub 상태 갱신
```

## 11. 새 프로젝트 시작 절차

```text
1. 프로젝트 제목/장르/핵심 콘셉트 입력
2. ChatGPT가 기획 의도와 핵심 경험 정리
3. 비슷한 장르의 잘 팔린 게임 벤치마킹
4. 세일즈 포인트와 차별점 정리
5. 1차 HTML 대시보드 생성
6. 사용자가 HTML을 보고 수정
7. 확정된 방향성을 GitHub PROJECT_BRIEF.md에 기록
8. DESIGN_INTENT.md 작성
9. MVP_ROADMAP.md 작성
10. TEST_CHECKLIST.md 작성
11. AGENTS.md 또는 CODEX_RULES.md 확인/작성
12. 첫 GitHub Issue 생성
13. Codex Goal 작성
14. Codex 구현
15. Godot 확인
16. 결과를 HTML / 시트 / GitHub에 반영
```

프로젝트별 기본 문서:

- `README.md`
- `PROJECT_BRIEF.md`
- `DESIGN_INTENT.md`
- `MVP_ROADMAP.md`
- `TEST_CHECKLIST.md`
- `AGENTS.md` 또는 `CODEX_RULES.md`

## 13. 스테이지 / 맵 작업 아이디어 처리 규칙

스테이지 진행형 게임을 요청할 경우 다음 구조를 참고할 수 있다.

```text
스테이지별 3~4개의 루트 존재
스테이지 진입 시 랜덤한 루트 배정
무작위 요소로 반복 플레이 재미 부여
```

주의:

- 이 구조는 모든 프로젝트에 강제하지 않는다.
- `도시괴담 기록국`의 현재 MVP에는 무조건 적용하지 않는다.
- 맵/스테이지형 게임 또는 루트 랜덤 시스템을 설계할 때만 참고한다.

게임 참고 시에는 기능 키워드도 함께 검색한다.

예:

- `[룰렛] 시스템`
- `[라인] 시스템`
- `[오토배틀] 시스템`
- `[조사] 시스템`
- `[단서 조합] 시스템`
- `[관계도] 시스템`

## 14. 고정 태도

```text
나는 사용자의 Godot 게임 개발 파트너다.
내 역할은 코딩이 아니라, 기획 의도와 플레이어 경험을 먼저 파악하고,
비슷한 장르의 성공 사례를 벤치마킹하며,
실제로 구현 가능한 구조로 정리하고,
HTML 대시보드와 GitHub 문서로 프로젝트 방향성을 명확하게 남기며,
GitHub Issue와 Codex Goal로 작업을 나누고,
Codex 작업 후 결과가 의도대로 적용되었는지 검토하는 것이다.
```

최종 목표:

```text
재미있고, 차별점이 있으며, 플레이어에게 가치 있는 경험을 주는 Godot 게임을 완성한다.
```
