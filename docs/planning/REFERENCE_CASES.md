# 괴이 기록국 적용 사례 라이브러리

> 문서 위치: `docs/planning/REFERENCE_CASES.md` | 공용 벤치마킹 방법: `docs/BENCHMARKING_REFERENCE_GUIDE.md` | 과거 조사 원문: `docs/benchmarks/`

## 1. 사용 목적

이 문서는 이미 프로젝트에서 사용한 벤치마킹과 내부 설계 사례를 **관찰 → 적용 → 제외 → 검증** 형식으로 압축한다.

새 작업자는 모든 과거 조사 문서를 다시 읽지 않고, 현재 질문과 맞는 사례만 선택한다. 외부 작품의 외형·문구·고유 규칙은 복제하지 않는다. 최신 제품 상태나 시장 정보가 중요한 판단에서는 원문과 최신 1차 출처를 다시 확인한다.

## 2. 사례 카드 형식

| 항목 | 의미 |
|---|---|
| 질문 | 해결하려던 실제 설계 문제 |
| 관찰 | 사례에서 확인한 구조와 사용자 가치 |
| 적용 | 괴이 기록국에 맞게 바꾼 원리 |
| 제외 | 장르·예산·프로젝트 정체성 때문에 쓰지 않은 요소 |
| 검증 | 실제 화면·데이터·플레이에서 확인할 항목 |
| 상태 | 채택, 조건부 채택, 보류, 폐기 |

## 3. 외부 벤치마킹 사례

### CASE-EXT-01 — 도시 장소와 인물 대화의 몰입

- **참고:** PARANORMASIGHT
- **질문:** 텍스트 중심 조사에서 도시 장소와 인물이 UI에 묻히지 않게 하려면?
- **관찰:** 배경, 인물, 대화가 같은 장면의 시선 흐름 안에 있을 때 장소 자체가 사건의 일부로 기억된다.
- **적용:** 조사·일상·후일담에서 배경과 캐릭터를 대시보드보다 먼저 보이게 하고, 현재 선택에 필요한 보조 UI만 주변부에 둔다.
- **제외:** 복수 관점 구조, 저주 규칙, 화면 외형과 연출 문법의 직접 복제.
- **검증:** 화면을 본 뒤 플레이어가 장소·현재 인물·지금 고를 질문을 순서대로 설명하는가?
- **상태:** 조건부 채택.

### CASE-EXT-02 — 관찰 근거를 스스로 조합하는 만족

- **참고:** The Case of the Golden Idol
- **질문:** 힌트를 제공하면서도 플레이어가 스스로 추론했다는 감각을 지키려면?
- **관찰:** 관찰 정보의 역할이 분리되고, 힌트가 정답을 직접 말하지 않을 때 ‘아하’ 경험이 생긴다.
- **적용:** 근거·힌트·위험 사례·분위기 기록을 분리하고, 보고서와 매뉴얼이 이미 본 근거를 다시 연결하게 한다.
- **제외:** 단어 끼워넣기 퍼즐, 별도 추리 문법, 기록장 UI 복제.
- **검증:** 플레이어가 힌트 문구가 아니라 두 개 이상의 관측 근거로 대응 이유를 설명하는가?
- **상태:** 채택.

### CASE-EXT-03 — 기록 재열람과 정보 은닉의 균형

- **참고:** Return of the Obra Dinn
- **질문:** 과거 기록을 다시 보는 재미를 주면서 메뉴 탐색 피로를 줄이려면?
- **관찰:** 추론을 갱신할 수 있는 기록 재열람은 강력하지만, 중요한 정보가 메뉴 깊숙이 숨으면 피로가 커진다.
- **적용:** 괴이 매뉴얼과 DB에서 확보한 기록을 재열람하고, 현재 판단과 직접 관련된 항목은 현장 화면에서 바로 연결한다.
- **제외:** 정보 위치를 의도적으로 숨기는 인터페이스와 고유 시각 스타일 모방.
- **검증:** 첫 사용자가 기록을 찾는 방법을 별도 설명 없이 이해하고, 같은 정보를 여러 화면에서 찾아 헤매지 않는가?
- **상태:** 조건부 채택.

### CASE-EXT-04 — 짧은 동료 반응과 학습 가능한 실패

- **참고:** Disco Elysium
- **질문:** 선택이 인물 관계에 기억되게 하면서 조사 흐름을 방해하지 않으려면?
- **관찰:** 동료의 짧은 반응과 실패 후 남는 의미가 선택을 기억하게 한다. 반면 장문 독백이 반복되면 현재 행동이 흐려진다.
- **적용:** 조사에서는 능력 발동·결정적 반응만 한 문장으로, 일상·후일담에서는 선택 기억을 말투와 행동 변화로 확장한다.
- **제외:** 스킬 인격화, 장문 내면 독백, 실패를 콘텐츠 소비량으로만 늘리는 구조.
- **검증:** 사건 뒤 플레이어가 요원의 수치보다 ‘어떻게 반응했는지’를 기억하는가?
- **상태:** 채택.

### CASE-EXT-05 — 대상 규칙과 요원 적합성

- **참고:** Lobotomy Corporation
- **질문:** 괴이 규칙과 요원 특성을 반복 학습하게 하면서 높은 손실 피로를 피하려면?
- **관찰:** 대상별 규칙과 요원 적합성을 학습하는 긴장감이 강하지만 영구 손실과 반복 비용은 진입 장벽이 된다.
- **적용:** 전조·요원 기여·체력·정신력의 이유를 보여주고, 실패는 위험 사례와 회복·후퇴 여지를 남긴다.
- **제외:** 시설 경영, 영구 사망, 과도한 반복 손실, 고유 작업명 복제.
- **검증:** 플레이어가 요원의 기여 이유를 이해하면서도 한 번의 실패로 장기 진행을 포기하지 않는가?
- **상태:** 채택.

### CASE-EXT-06 — 선택·일정·관계가 장기 서사에 남는 구조

- **참고:** 활협전, 화산의 딸, 서울 2033, The Winter Tower
- **질문:** 짧은 텍스트 장면과 장기 캠페인 선택을 어떻게 연결할 것인가?
- **관찰:** 짧은 선택이 일정·관계·후속 사건에 남을 때 반복 플레이의 의미가 생긴다. 과도한 RNG·가혹한 영구 처벌·방대한 육성 수치는 진입 부담이 된다.
- **적용:** 반일을 소모하지 않는 일상 장면, 작은 선택 기억, 후속 에피소드 해금, DB 기록으로 연결한다.
- **제외:** 대규모 육성 시뮬레이션, 연애 수치 중심, 영구 처벌, 선택 결과를 읽기 어려운 과도한 랜덤.
- **검증:** 선택 콘텐츠를 보지 않아도 핵심 진행이 가능하고, 본 플레이어는 다음 대사와 장면 차이를 체감하는가?
- **상태:** 조건부 채택.

## 4. 프로젝트 내부 사례

### CASE-INT-01 — 대시보드형 조사 화면을 장면 중심으로 재구성

- **발생 범위:** MVP-043 저승역 UI
- **문제:** 정보 패널이 많아 배경·현재 상황·첫 선택이 한눈에 읽히지 않음.
- **개선:** 장소, 판단, 페이지형 괴이 매뉴얼을 3열로 정리하고, 실패 후보를 지우지 않고 배제 근거와 함께 남김.
- **재사용 원칙:** UI를 줄이는 것보다 현재 질문의 시선 순서를 정하고, 상세 기록은 필요할 때 열게 한다.
- **주의:** 다른 사건에 같은 3열 외형을 기계적으로 복제하지 않고 현재 판단 역할을 기준으로 적용한다.
- **검증:** 1280×720에서 현재 상황·근거·첫 선택이 스크롤 없이 보이는가?

### CASE-INT-02 — 안내자 이름과 내부 ID 분리

- **발생 범위:** `로그` → `기록관 아카`
- **문제:** 플레이어에게 보이는 캐릭터 정체성을 바꾸면서 내부 저장 키와 파일명을 일괄 치환하면 회귀 위험이 큼.
- **개선:** 플레이어 표시명과 대사는 `기록관 아카`로 통일하고, 내부 `로그` ID·파일명·저장 키는 호환용으로 유지.
- **재사용 원칙:** 표시 계층과 영속 상태 식별자를 분리한다.
- **주의:** 새 문서가 내부 키를 잘못된 공식 명칭으로 다시 정의하지 않게 한다.
- **검증:** 저장 왕복과 기존 튜토리얼 키가 유지되면서 플레이어 화면에는 구명칭이 노출되지 않는가?

### CASE-INT-03 — 조사 대사와 일상 대사의 밀도 분리

- **발생 범위:** MVP-044 기획
- **문제:** 캐릭터 매력을 늘리려다 조사 흐름이 장문 대화로 느려질 위험.
- **개선:** 조사에서는 상황지시문·선택·아카 정리 중심, 요원 대사는 기능적 한 문장으로 제한. 일상·후일담에서 대화와 감정 변화를 충분히 제공.
- **재사용 원칙:** 콘텐츠 유형별 대사 예산을 분리하고 같은 인물의 말투는 유지한다.
- **검증:** 조사 대사를 가려도 선택 흐름이 이해되고, 일상에서는 말투만으로 인물을 구분하는가?

### CASE-INT-04 — 숫자형 호감도 대신 선택 기억

- **발생 범위:** MVP-045 기획
- **문제:** 요원을 연애 호감도나 성능 보너스 카드로 축소할 위험.
- **개선:** 신뢰·경계·부채·경쟁·공감·보호·균열 태그를 대사·정보 공유·장면 해금으로 표현.
- **재사용 원칙:** 내부 계산과 플레이어 경험을 분리하고, 관계 변화는 행동으로 보여준다.
- **주의:** 기존 신뢰 상태를 즉시 삭제하지 않고 저장 호환과 파생 가능성을 먼저 감사한다.
- **검증:** 플레이어가 수치 없이도 관계 변화를 설명하는가?

### CASE-INT-05 — 프레젠테이션 계층의 상태 비소유

- **발생 범위:** MVP-046 기획
- **문제:** 표정·컷인·대화 UI가 별도 분기와 저장 상태를 소유하면 사건·관계 데이터가 중복됨.
- **개선:** 서사 데이터가 감정 의미 키와 컷인 종류를 요청하고, 공용 프레젠테이션 계층은 자산을 선택해 재생한 뒤 폴백한다.
- **재사용 원칙:** 연출은 상태를 표현하지만 결정하지 않는다.
- **검증:** 자산이 없거나 연출을 끈 상태에서도 동일한 선택·결과·저장 상태가 유지되는가?

### CASE-INT-06 — 문서 원본과 백업 분리

- **발생 범위:** 2026-07-16 문서 정리
- **문제:** 오래된 상태·완료 QA·중복 기획이 기본 읽기에 남아 새 작업마다 불필요한 확인이 반복됨.
- **개선:** `CURRENT_STATUS.md`를 단일 상태 원본으로 만들고, 조건부 문서 라우터와 날짜별 백업을 도입. 기존 링크는 짧은 리디렉션으로 유지.
- **재사용 원칙:** 한 주제 한 원본, 구현 사실과 승인 계획 분리, 과거 근거는 삭제하지 않고 기본 읽기에서 제외.
- **검증:** 새 작업자가 기본 3~4개 문서만 읽고 현재 구현과 다음 계획을 구분하는가?

### CASE-INT-07 — 생성 이미지의 텍스트 분리

- **발생 범위:** MVP-043 캐릭터 아트
- **문제:** 생성 이미지 안의 한글과 카드 UI는 오탈자·품질·현지화·수정 비용이 높음.
- **개선:** 아트는 텍스트 없는 캐릭터·장면 자산으로 제작하고, 이름·소속·능력·대사는 게임 UI가 별도 렌더링.
- **재사용 원칙:** 이미지 모델은 시각 자산, 엔진은 의미 텍스트와 인터랙션을 담당한다.
- **검증:** 자산에 구운 텍스트가 없고 해상도·알파·실루엣이 manifest와 일치하는가?

## 5. 사례 사용 절차

1. 현재 설계 질문을 한 문장으로 적는다.
2. 위 사례 중 같은 문제를 다룬 카드 1~3개만 선택한다.
3. 관찰 원리와 현재 프로젝트 제약을 분리한다.
4. 적용·조건부 적용·제외를 기록한다.
5. 화면·대사·저장·플레이테스트 검증으로 결정한다.
6. 새로운 반복 가능 원칙은 프로젝트 특화 요소를 제거해 Base 승격 후보로 정리한다.

## 6. 새 사례 추가 조건

다음 항목이 모두 있어야 사례 라이브러리에 추가한다.

- 실제 해결한 질문
- 관찰한 출처 또는 프로젝트 상황
- 적용한 변경
- 적용하지 않은 요소와 이유
- 검증 결과 또는 아직 검증되지 않았다는 상태
- 다른 작업에 재사용 가능한 원칙

단순 감상, 작품 목록, 출처 없는 일반론은 사례로 추가하지 않는다.
