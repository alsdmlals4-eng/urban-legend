# MVP Status Audit

## 점검 일자

2026-07-10

## 점검 목적

MVP-022 기준으로 조사 화면 UI와 단서/힌트 추적 구조를 확인하고, PC/Steam 기준 조사 화면 정보 위계 개선 방향을 정리한다.

## 결론 요약

현재 큰 방향의 충돌은 없다.

MVP-022는 새 괴담 에피소드 추가가 아니라 `조사 화면 UI / 단서·힌트 추적 개선 1차`로 진행한다.

프로젝트의 1차 출시 목표는 **PC 기준 Steam 출시**다. 따라서 UI 판단은 PC 16:9, 마우스/키보드, Steam 데모 첫인상 기준을 우선한다.

핵심 판단:

```text
조사 포인트를 누른 결과가 지나가는 텍스트가 아니라,
사건 기록으로 축적되고 다음 판단에 쓰인다는 느낌을 줘야 한다.
```

## MVP-022 UI 진단

| 화면/요소 | 현재 상태 | 개선 방향 |
|---|---|---|
| 조사 화면 전체 | 상태, 조사 포인트, 결과, 힌트, 단서 목록이 한 스크롤 흐름에 있음 | 사건 상태/조사 포인트/결과/단서/힌트/회수 안내를 섹션화 |
| 조사 포인트 | 버튼 목록으로 표시, 잠김 상태는 텍스트 prefix 사용 | 잠김 이유와 클릭 가능 상태를 더 명확히 표시 |
| 조사 결과 | 결과 텍스트가 `_result_label` 중심으로 표시 | 이번 조사 결과와 새 정보가 로그처럼 남는 느낌 강화 |
| 조사 방법 결과 | 판정식, 상태 변화, 신뢰도, 이벤트가 한꺼번에 출력 | 요약 먼저, 상세 판정식은 보조 정보로 낮춤 |
| 단서 목록 | 수집됨/미수집 + 제목 중심 | 수집/미수집/이번 획득 단서와 단서 의미를 구분 |
| 힌트 | `_hint_label`로 별도 표시되지만 목록 추적은 약함 | 단서 목록과 분리된 힌트 목록/연결 문구 강화 |
| 회수/해결 버튼 | `해결 시도` 중심 표현 | 회수/안정화 조건 표현을 강화하되 함수명 대규모 변경은 피함 |

## Base 승격 후보

- 조사 화면은 결과 텍스트보다 `새 정보 / 상태 변화 / 다음 행동`을 먼저 보여줘야 한다는 원칙
- 단서와 힌트는 같은 정보가 아니므로 UI에서도 분리해야 한다는 원칙

## 프로젝트 전용으로 유지할 내용

- PC 기준 Steam 출시 목표
- Unity Asset Store / Tales of Tuscany Demo 고정 참고 링크
- 기록국 DB / 조사 포인트 / 단서 / 힌트 / 회수 표현

## 남은 위험

- 실제 Godot 화면에서 PC 16:9, 창 모드, 전체화면 기준 가독성은 로컬 실행으로 확인해야 한다.
- `investigation_scene.gd`가 코드 생성형 UI이므로 레이아웃 수정 중 신호 연결이 깨질 수 있다.
- MVP-023 사건 보고서/DB UI 개선과 범위가 섞이지 않도록 한다.
