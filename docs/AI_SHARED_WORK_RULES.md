# AI Shared Work Rules

> 수명주기: `COMPATIBILITY_STUB`이 포함된 보존 문서다. 최신 작업 계약은 `docs/OPERATING_MODEL.md`와 `docs/WORK_MODE_AND_SKILL_ROUTING.md`가 책임진다. 아래 원문은 과거 링크와 고유 절차 보존용이며, 충돌 시 최신 사용자 지시 → `AGENTS.md` → 최신 운영 원본 순으로 판단한다.

> **Base 공용 규칙의 로컬 사본**: 원본은 [`alsdmlals4-eng/Base`](https://github.com/alsdmlals4-eng/Base)이며, 기준 커밋과 동기화 방식은 `docs/BASE_RULES_VERSION.md`를 따른다. 일반 작업에서는 Base 링크보다 이 로컬 사본을 먼저 읽는다.

이 문서는 특정 엔진이나 장르에 종속되지 않는 공용 AI 작업 규칙이다. 새 프로젝트를 시작할 때 복사하고, 프로젝트의 `AGENTS.md`와 전용 규칙 문서가 이 문서보다 구체적인 내용을 덧붙인다.

## 1. 규칙 우선순위

충돌 시 다음 순서로 판단한다.

1. 사용자의 최신 지시
2. 프로젝트 루트의 `AGENTS.md` 또는 `CODEX_RULES.md`
3. 현재 GitHub Issue와 승인된 설계 문서
4. 현재 작업의 Goal 또는 PR 설명
5. 이 공용 규칙
6. 과거 대화와 추측

규칙과 실제 파일이 다르면 실제 파일을 확인하고 차이를 보고한다. 정상 동작 중인 사용자 변경사항은 되돌리지 않는다.

## 2. 역할 분리

- 기획/조정 AI: 사용자 의도, 플레이어 가치, 벤치마킹, Issue, Goal, 대시보드, 검수 기준을 정리한다.
- 구현 AI: 승인된 Issue/Goal을 기준으로 실제 파일을 읽고 수정하고 검증한다.
- 사용자: 방향을 결정하고, 실제 실행 결과와 플레이 감각을 확인하고 승인한다.

한 작업에서 역할을 섞지 않는다. 구현 AI가 기획을 임의로 확장하지 않고, 기획 AI가 구현 완료를 추정하지 않는다.

## 3. 공용 문서와 프로젝트 전용 문서 구분

새 프로젝트는 `공용 규칙 패키지`와 `프로젝트 전용 기억`을 분리한다.

### 공용 규칙 패키지

여러 프로젝트에 재사용할 수 있는 문서다.

- `docs/AI_SHARED_WORK_RULES.md`: 공용 AI 작업 원칙
- `docs/AI_WORKFLOW_RULES.md`: ChatGPT 작업 순서와 Goal 작성 규칙
- `docs/AI_SKILL_ADOPTION_GUIDE.md`: 외부 스킬 채택, 권한 사전점검, context compact 기준
- `docs/BENCHMARKING_REFERENCE_GUIDE.md`: 벤치마킹 표 형식과 사례 관리 방식
- `docs/MVP_WORKFLOW_CHECKLIST.md`: 실제 작업 시작/종료 체크리스트
- `docs/DOCUMENTATION_MAP.md`: 문서 역할표와 공용/전용 구분

주의:

- 루트 `AGENTS.md`는 보통 프로젝트 전용 문서다.
- 공용 AGENTS 내용은 이 문서와 공용 템플릿에 둔다.
- 새 프로젝트에서는 공용 원칙을 복사한 뒤, 해당 프로젝트의 엔진·폴더·금지 범위를 `AGENTS.md`에 새로 적는다.

### 프로젝트 전용 기억

프로젝트마다 새로 작성하거나 수정한다.

- `AGENTS.md`: 프로젝트 최상위 규칙
- `docs/CODEX_SHARED_WORK_RULES.md` 또는 `docs/CODEX_PROJECT_RULES.md`: 프로젝트용 Codex 구현 규칙
- `README.md`: 소개, 실행 방법, 현재 상태
- `PROJECT_BRIEF.md`: 한 줄 설명, 장르, 플레이어 역할, 핵심 경험, 차별점
- `DESIGN_INTENT.md`: 기획 의도, 핵심 루프, 플레이어 감정, 선택지 설계 원칙
- `MVP_ROADMAP.md`: MVP 목록, Issue 번호, 상태, 완료 기준, 다음 작업
- `TEST_CHECKLIST.md`: 실행 검수 순서, MVP별 체크리스트, 오류 기록 방식
- `.serena/project.yml`: 프로젝트별 Serena 설정
- `data/`: 게임 데이터
- `scripts/`: 구현 코드
- `scenes/`: 씬 파일

## 4. 전용 프로젝트 설계 절차

새 프로젝트를 시작하면 다음 순서로 설계한다.

1. 프로젝트 제목, 장르, 핵심 콘셉트를 정한다.
2. 벤치마킹할 장르와 인접 사례를 정한다.
3. 핵심 플레이어 경험을 한 문장으로 정의한다.
4. 반복 루프를 정의한다.
5. 만들지 않을 것을 먼저 정해 범위 폭발을 막는다.
6. `PROJECT_BRIEF.md`를 작성한다.
7. `DESIGN_INTENT.md`를 작성한다.
8. `MVP_ROADMAP.md`를 작성한다.
9. `TEST_CHECKLIST.md`를 작성한다.
10. 프로젝트용 `AGENTS.md`와 Codex 규칙 문서를 작성한다.
11. 첫 GitHub Issue를 만들고 Goal을 작성한다.

```md
## 핵심 경험

## 핵심 루프

## 차별점

## 제외할 것

## 현재 포커스

## 다음 작업
```

### DESIGN_INTENT.md

```md
# DESIGN_INTENT

## 기획 의도

## 플레이어에게 주고 싶은 감정

## 핵심 선택 구조

## 벤치마킹 반영 원칙

## 금지 방향
```

### AGENTS.md

```md
# AGENTS

## 기본 규칙

- 이 프로젝트의 엔진과 언어를 명시한다.
- 다른 엔진 기준 설명을 금지한다.
- 기존 사용자 변경사항을 되돌리거나 덮어쓰지 않는다.
- 요청 범위 밖 리팩터링을 하지 않는다.
- 정상 작동하는 구조를 임의로 바꾸지 않는다.
- 작업 전 README와 프로젝트 브리프를 확인한다.
- 작업 후 변경 파일, 구현 방식, 검증 결과, 남은 위험을 요약한다.

## 작업 기준

모든 구현 작업은 GitHub Issue 또는 승인된 Goal을 기준으로 한다.
Issue에 없는 기능은 추가하지 않는다.
```

## 7. 작업 유형별 경로

### 빠른 경로

오탈자, 단순 문서, 이미 결정된 단일 데이터 수정은 다음만 수행한다.

`최신 지시 확인 → 실제 파일 확인 → 범위 밖 변경 여부 확인 → 수정 → 최소 검증 → 보고`

### 기능 경로

새 시스템, 여러 파일, 저장/불러오기, UI/UX, 게임 루프 변경은 다음을 수행한다.

`의도 분석 → 기준 문서/Issue 확인 → 준비 상태 확인 → 필요한 근거 조사 → 범위·위험 확정 → Issue → Goal → 구현 → 검증 → 회고`

불확실한 사실이나 현재 바뀔 수 있는 정보는 웹 또는 공식 문서로 확인한다. 이미 확정된 구현을 그대로 옮기는 작업에는 불필요한 조사를 반복하지 않는다.

## 8. Definition of Ready

구현을 시작하기 전에 다음 조건을 만족해야 한다.

- [ ] 한 문장 목표와 해결하려는 사용자 문제가 있다.
- [ ] 이전 작업과 다음 작업의 연결이 설명된다.
- [ ] 구현 범위와 제외 범위가 분리되어 있다.
- [ ] 완료 기준이 관찰 가능한 행동 또는 검증 가능한 상태로 작성되어 있다.
- [ ] 예상 영향 파일과 저장/호환성 위험이 있다.
- [ ] 실행·검증 방법과 사용자가 확인할 순서가 있다.
- [ ] 필요한 경우 벤치마킹의 반영/제외 결론이 있다.

준비되지 않은 요구사항은 구현자에게 넘기지 말고 질문, Issue 보완, 또는 작은 탐색 작업으로 분리한다.

## 9. 근거 조사와 벤치마킹

- 새 시스템이나 방향 결정은 관련 사례 3~5개를 먼저 조사하고, 각 사례에서 관찰점·반응·우리 프로젝트 적용 이유를 기록한다.
- 장르 핵심 구조, 시장성 판단, 큰 방향 전환처럼 표본이 중요한 경우에만 10개 이상으로 확장한다.
- 조사는 20~30분 단위로 시간 제한하고, 사례 수보다 적용 결론을 우선한다.
- 공식 문서, 개발자 발언, 학술/업계 자료, 실제 사용자 반응을 구분한다.
- 출처와 사실을 남기되, 다른 작품의 기능을 그대로 복사하지 않는다.
- 구현이 이미 Issue에서 확정된 경우에는 벤치마킹을 새로 강제하지 않는다.

## 10. Issue, Goal, PR

- Issue는 장기 기준서다: 문제, 목표, 흐름, 범위, 제외, 완료 기준, 위험, 검수 항목을 둔다.
- Goal은 구현 지시서다: Issue와 전용 규칙을 참조하고, 구현 범위와 완료 기준만 짧게 둔다.
- PR은 변경 이유, 영향, 검증 결과, 남은 위험을 기록하는 변경 이력이다.
- 한 Issue는 한 가치 흐름을 설명하고, 관련 없는 개선은 별도 Issue로 분리한다.
- 완료 기준은 기능명보다 `조건 → 행동 → 관찰 결과` 형태로 쓴다.

## 11. 구현 전후 규칙

구현 전에는 현재 브랜치와 작업 상태, 관련 파일, 호출/참조 위치, 저장 포맷을 확인한다. 수정 전에는 대상 파일과 변경 계획을 짧게 보고한다.

구현 후에는 다음을 확인한다.

- [ ] 실제 변경 파일과 diff가 Issue 범위 안이다.
- [ ] 문법, 데이터 형식, 핵심 실행 경로를 검증했다.
- [ ] 저장/불러오기와 기존 흐름에 미치는 영향을 확인했다.
- [ ] 테스트하지 못한 항목을 숨기지 않았다.
- [ ] 사용자가 실행할 검수 순서를 적었다.
- [ ] 남은 위험을 수동 확인이 필요한 것과 수정 가능한 것으로 구분했다.

재현 가능한 작은 위험은 보고로 넘기지 말고 현재 작업에서 먼저 줄인다. 범위를 넘는 개선은 별도 제안으로 남긴다.

### 서사형 안내자와 튜토리얼 정보 경계

- 서사형 튜토리얼 안내자는 담당 시스템과 자연스럽게 연결되는 세계관 내 역할을 가진다.
- 최초 상세 안내와 재방문용 짧은 안내를 분리하고 확인 상태를 저장한다.
- 음향 신호는 음소거 상태에서도 이해할 수 있는 동등한 시각 또는 문구 신호와 함께 제공한다.
- 안내자는 플레이어가 아직 획득하지 않은 단서, 정답, 숨은 수치나 결과를 누설하지 않는다.

## 12. GitHub 작업 단위

- 작은 변경은 목적이 하나인 커밋으로 묶고, 커밋 제목은 `type: concise purpose` 형식으로 쓴다.
- 구현 완료와 배포/푸시는 별개다. 사용자의 명시적 요청 없이 외부에 push, Issue 댓글, PR 생성 등을 하지 않는다.
- PR을 사용하는 팀은 Issue 링크, 검증 결과, 수동 확인 항목을 PR 본문에 남긴다.
- 자동 검사나 필수 리뷰가 없는 저장소는 이를 가정하지 말고 실제 설정을 보고한다.

## 13. 회고와 재사용

작업이 끝나면 다음 세 가지를 기록한다.

- 반복된 혼동 또는 실패
- 다음 작업에서 줄이거나 추가할 규칙
- 공용 규칙으로 승격할 내용과 프로젝트 전용으로 남길 내용

공용 규칙에는 안정적인 원칙만 넣고, 엔진·폴더·게임 세계관·일회성 함수명은 프로젝트 전용 문서에 둔다.
