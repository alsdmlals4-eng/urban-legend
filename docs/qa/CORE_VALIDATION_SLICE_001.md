# CORE-VALIDATION-001 저승역 가설·근거 계약

> 상태: `IMPLEMENTED_1A / UI_1B_PENDING`
> 대상 브랜치: `agent/core-validation-slice`
> 기준 구현: `main` MVP-043 / 저장 Schema `mvp-039`

## 1. 목적

저승역 회수 단계에서 플레이어가 능력치·확률·자동 예측만 보고 정답을 고르지 않고, 현재 전조와 확보 기록을 비교해 가설과 대응 이유를 읽도록 만든다.

이번 단계는 전체 회수 UI를 한 번에 교체하지 않는다. 먼저 기존 네 패턴의 행동 계약을 보존하면서 가설·지지 근거·반증 근거·추론 문장을 데이터로 정의하고, 자동 예측 문구가 정답 행동을 직접 설명하지 않도록 한다.

## 2. 범위

### 1A — 이번 PR

- 저승역 전용 선택적 데이터 오버레이 로더.
- 네 회수 패턴의 가설·근거·반증·추론 계약.
- 자동 예측에 노출되는 `name`·`description`의 정답 누설 완화.
- 기존 패턴 ID·순서·전조·관련 단서·대응 ID·능력·안정도 획득량·정답 ID 보존.
- Python 표준 라이브러리 기반 계약 테스트와 전용 GitHub Actions.

### 1B — 후속 PR

- 회수 화면에서 `가설 선택 → 근거 선택 → 대응 선택`을 명시적으로 분리한다.
- 가설과 근거를 선택하기 전에는 대응 버튼을 노출하지 않는다.
- 잘못된 가설도 즉시 정답으로 교정하지 않고, 대응 결과 뒤 반증 근거와 위험 사례를 기록한다.
- 가설·근거 선택은 UI 파생 상태로 시작하며 저장 Schema를 늘리지 않는다.

### 이번 범위에서 하지 않는 것

- `scripts/core/game_state.gd` 변경.
- 저장 Schema 변경 또는 새 저장 필드 추가.
- 저승역 원본 JSON 삭제·이동·ID 변경.
- 다른 두 사건의 패턴 변경.
- 능력치·HP·정신력·시장·장비·세력 기능 삭제.
- 미니게임 규칙 변경.

## 3. 플레이어 질문

각 전조는 다음 순서를 가져야 한다.

```text
현재 전조 확인
→ 확인할 질문 읽기
→ 가능한 규칙 가설 비교
→ 지지 또는 반증 기록 확인
→ 대응 선택
→ 현장 반응 확인
→ 성공 근거 또는 위험 사례 기록
```

자동 예측 성공은 위 순서를 생략하지 않는다. 예측은 현재 비교 질문을 정리할 수 있지만 정답 대응을 직접 지시하지 않는다.

## 4. 데이터 계약

파일: `data/episodes/episode_001_afterlife_station_core_validation.json`

### 오버레이 루트

| 필드 | 의미 |
|---|---|
| `target_episode_id` | 적용할 사건 ID. 원본 사건 ID와 다르면 적용하지 않는다. |
| `contract_version` | 코어 검증 계약 버전. |
| `purpose` | 작성 목적. 런타임 상태가 아니다. |
| `overrides` | 원본 최상위 키를 대체할 데이터. 이번 단계에서는 `recovery_patterns`만 허용한다. |

### 패턴 필수 필드

| 필드 | 의미 |
|---|---|
| `id`, `order` | 기존 식별자와 작성 순서. 변경 금지. |
| `telegraph` | 플레이어가 직접 관측하는 현상. 원본 보존. |
| `description` | 자동 예측에 노출 가능한 중립적 비교 과제. 정답 행동을 말하지 않는다. |
| `question` | 현재 전조에서 판별할 한 가지 질문. |
| `manual_draft` | 매뉴얼에 남길 수 있는 임시 규칙 문장. |
| `related_clue_ids` | 현재 질문과 직접 연결된 기존 단서. 원본 보존. |
| `responses` | 기존 대응 ID와 능력·안정도 계약을 유지한 가설형 대응. |
| `correct_response_id` | 기존 정답 ID. 변경 금지. |
| `failure_reason` | 오대응이 어떤 근거와 충돌했는지 설명한다. |

### 대응 필수 필드

| 필드 | 의미 |
|---|---|
| `hypothesis` | 해당 대응이 전제로 삼는 규칙 가설. |
| `supporting_clue_ids` | 가설을 지지하는 확보 기록 후보. |
| `contradicted_clue_ids` | 가설을 반박하는 확보 기록 후보. |
| `reasoning` | 지지·반증이 대응 선택으로 이어지는 이유. |
| `summary` | 현재 ActionChoiceCard 툴팁과 후속 UI에서 사용할 짧은 비교 문구. |

정답 대응은 패턴의 모든 `related_clue_ids`를 지지 근거로 가져야 한다. 오답 대응은 최소 하나의 반증 근거를 가져야 한다. 지지와 반증에 같은 단서를 동시에 넣지 않는다.

## 6. 로더 안전 경계

파일: `scripts/data/episode_loader.gd`

1. 기본 사건 JSON을 먼저 읽는다.
2. 파일명이 `.json`일 때만 같은 경로의 `_core_validation.json`을 확인한다.
3. 오버레이가 없으면 기존 Dictionary를 그대로 반환한다.
4. `target_episode_id`가 원본 사건 ID와 다르면 오류를 남기고 기존 Dictionary를 반환한다.
5. `overrides`가 Dictionary가 아니면 오류를 남기고 기존 Dictionary를 반환한다.
6. 원본을 `duplicate(true)`한 뒤 지정된 최상위 키만 대체한다.
7. 오버레이 파일 자체를 직접 로드했을 때 재귀 적용하지 않는다.

## 7. 적대적 검토

### 공격 1 — 오버레이가 다른 사건을 오염시킬 수 있다

- 방어: 파일명 규칙, 대상 사건 ID 검증, 오버레이 부재 시 원본 반환.
- 잔여 위험: 동일 사건 ID를 잘못 복제한 파일은 통과할 수 있다. 파일 경로와 ID를 CI에서 함께 검수한다.

### 공격 2 — 가설 데이터가 기존 정답·밸런스를 바꿀 수 있다

- 방어: 패턴·대응·정답·능력·안정도 획득량을 원본과 자동 비교한다.
- 잔여 위험: 문구가 플레이어에게 지나치게 명백하거나 모호할 수 있다. 1B 실화면 QA와 신규 플레이테스트가 필요하다.

### 공격 3 — 자동 예측이 여전히 정답을 대신할 수 있다

- 방어: 예측이 표시하는 패턴명과 설명을 중립 질문으로 바꿨다.
- 잔여 위험: 현재 `roll_anomaly_prediction`은 성공 시 패턴 설명을 강조한다. 1B에서 가설 UI가 앞에 오도록 정보 위계를 재배치해야 한다.

### 공격 4 — 새 필드가 런타임에서 사용되지 않아 죽은 데이터가 될 수 있다

- 판정: `SHOULD_FIX / 1B REQUIRED`.
- 방어: 이번 PR을 독립 완성 기능으로 선언하지 않고 `IMPLEMENTED_1A / UI_1B_PENDING`으로 표시한다. 계약 테스트를 먼저 고정해 다음 UI 구현이 임의 해석하지 못하게 한다.

### 공격 5 — 저장 중간 상태가 추가될 수 있다

- 방어: 이번 단계는 GameState와 저장 파일을 변경하지 않는다. 1B도 우선 UI 파생 상태로 구현하고, 회수 대응 결과만 기존 방식으로 저장한다.
