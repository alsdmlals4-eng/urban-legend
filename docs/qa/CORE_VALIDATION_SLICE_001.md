# CORE-VALIDATION-001 저승역 가설·근거 검증 슬라이스

> 상태: `IMPLEMENTED_1A_1B_1C / GODOT_VALIDATED / DOCS_SYNCED / PLAYER_TEST_PENDING`
> 적용 기준: 최신 `main` 기반 clean PR. 실제 브랜치·커밋은 PR 메타데이터를 따른다.
> 기준 구현: `MVP-043 + CORE-VALIDATION-001` / `Ver 4.2` / 저장 Schema `mvp-039`
> 검증 엔진: Godot `4.7.1.stable.official.a13da4feb`

## 1. 목적

저승역 회수 단계에서 플레이어가 능력치·확률·자동 예측만 보고 정답을 고르지 않고 다음 판단을 직접 수행하게 한다.

```text
현재 전조 관측
→ 규칙 가설 선택
→ 확보 기록 선택
→ 가설에 따른 대응 선택
→ 현장 결과 확인
→ 검증된 근거 또는 위험 사례 기록
```

현장 안정화와 매뉴얼 검증은 분리한다. 기존 정답 대응을 실행하면 안정화 수치는 그대로 적용되지만, 가설과 작성된 모든 지지 근거가 일치하지 않으면 매뉴얼 후보는 `미검증` 또는 `근거 일부`로 남는다.

## 2. 구현 범위

### 1A — 데이터·로더 계약

- 저승역 전용 선택적 `_core_validation.json` 오버레이.
- 네 회수 패턴의 질문·가설·지지 근거·반증 근거·추론·매뉴얼 초안.
- 자동 예측에 노출되는 패턴명·설명을 중립적 비교 과제로 수정.
- 기존 패턴·대응·정답·밸런스 계약 자동 비교.
- 잘못된 오버레이 또는 타 사건 ID에서는 원본 사건 데이터를 반환.

### 1B — 실제 회수 UI

- `가설 → 근거 → 대응` 3단계 UI.
- 가설 선택 전에는 근거와 대응을 노출하지 않는다.
- 근거는 현재 확보한 연결 단서만 선택할 수 있다.
- 연결 단서를 하나도 확보하지 못한 경우 기존 진행을 막지 않고 `근거 없이 대응`을 허용한다.
- `Esc`와 이전 단계 버튼으로 대응→근거→가설 역방향 이동.
- 근거 단계로 돌아가도 선택 기록을 유지한다.
- 각 단계의 첫 사용 가능한 카드로 키보드 포커스를 이동한다.
- 비대상 사건은 기존 직접 대응 UI로 폴백한다.
- 가설·근거 선택은 회수 화면의 파생 상태이며 새 저장 필드를 만들지 않는다.

### 1C — 괴이 매뉴얼 승격·저장·재열람

- `anomaly_manual_records`를 기존 `mvp-039` 저장 payload의 선택 Dictionary로 추가한다.
- 작성된 모든 지지 근거까지 확인한 성공만 `verified_rules`의 공식 규칙으로 승격한다.
- 대응은 맞았지만 가설·근거가 불충분하면 `candidate_rules`에 검증 대기 후보로 남긴다.
- 오대응은 `danger_cases`에 남기며, 같은 패턴·대응·가설의 반복은 중복 행을 만들지 않고 시도 횟수만 증가시킨다.
- 후보가 공식 규칙으로 승격되면 같은 패턴 후보를 제거하되 과거 위험 사례는 보존한다.
- 이미 공식 규칙인 패턴을 근거 부족 상태로 다시 시도해도 공식 규칙을 후보로 강등하지 않는다.
- 결과 화면과 기록국 DB는 정적 저승역 예시가 아니라 현재 저장된 공식 규칙·후보·위험 사례를 표시한다.
- 완료 사건 보고서는 당시 `anomaly_manual_record` 스냅샷을 포함해 이후 재열람 시 판단 근거를 보존한다.
- 기존 저장에 필드가 없으면 빈 Dictionary로 이관하며 `SAVE_VERSION`은 `mvp-039`를 유지한다.

### 다음 단계

- **1D:** 신규 플레이어가 규칙과 대응 이유를 자기 말로 설명하는지 관찰.

## 3. 플레이어 정보 경계

### 선택 전에 보여 주는 정보

- 직접 관측한 전조.
- 중립적인 판단 질문.
- 가능한 규칙 가설.
- 이미 확보한 사건 기록의 제목과 내용.
- 대응 행동의 문구와 실행 보조 능력 종류.

### 선택 전에 보여 주지 않는 정보

- 정답 대응 ID.
- 최적 요원 이름과 능력 수치.
- 가설이 정답인지 오답인지 나타내는 `일치/불일치` 표식.
- 데이터의 `reasoning`, `summary`, `반증 후보` 문구.
- 미확보 단서의 상세 내용.

`summary`와 `reasoning`은 선택 전 툴팁으로 사용하지 않는다. 현장 결과와 매뉴얼 기록을 설명하는 후행 정보로만 사용한다.

## 4. 데이터 계약

파일: `data/episodes/episode_001_afterlife_station_core_validation.json`

### 오버레이 루트

| 필드 | 의미 |
|---|---|
| `target_episode_id` | 원본 사건 ID와 다르면 적용하지 않는다. |
| `contract_version` | 코어 검증 계약 버전. |
| `purpose` | 작성 목적. 런타임 상태가 아니다. |
| `overrides` | 대체할 최상위 데이터. 현재 `recovery_patterns`만 허용한다. |

### 패턴 필수 필드

| 필드 | 의미 |
|---|---|
| `id`, `order` | 기존 식별자와 순서. 변경 금지. |
| `telegraph` | 플레이어가 직접 관측하는 기존 전조. |
| `description` | 자동 예측에서 사용할 중립적 비교 과제. |
| `question` | 전조에서 판별할 한 가지 질문. |
| `manual_draft` | 검증 뒤 매뉴얼 후보가 될 임시 규칙. |
| `related_clue_ids` | 질문과 직접 연결된 기존 단서. |
| `responses` | 기존 대응 계약 위에 가설·근거를 추가한 목록. |
| `correct_response_id` | 기존 정답 ID. 변경 금지. |
| `failure_reason` | 오대응과 기록이 충돌하는 이유. |

### 대응 필수 필드

| 필드 | 의미 |
|---|---|
| `hypothesis` | 대응이 전제로 삼는 규칙 가설. |
| `supporting_clue_ids` | 가설을 지지하는 작성 근거 전체. |
| `contradicted_clue_ids` | 가설을 반박하는 작성 근거. |
| `reasoning` | 결과 뒤 근거와 대응의 연결을 설명한다. |
| `summary` | 결과·매뉴얼용 짧은 비교 기록. 선택 전 정답 표식으로 사용하지 않는다. |

정답 가설은 패턴의 모든 `related_clue_ids`를 지지 근거로 가져야 한다. 오답 가설은 최소 하나의 반증 근거를 가져야 한다. 같은 단서를 지지와 반증에 동시에 넣지 않는다.

## 5. 상태와 판정

### UI 파생 상태

```text
DIRECT      비대상 사건의 기존 대응
HYPOTHESIS  가설 선택
EVIDENCE    확보 기록 선택
RESPONSE    현장 대응 선택
```

다음 값은 `battle_scene.gd`가 현재 턴 동안만 소유한다.

- 선택 가설의 대응 ID.
- 선택 근거 ID 배열.
- 현재 의사결정 단계.

씬 전환이나 새 전조 시작 시 초기화하며 저장 Schema에는 포함하지 않는다.

### 안정화와 매뉴얼 검증 분리

| 조건 | 현장 안정화 | 매뉴얼 판정 |
|---|---|---|
| 정답 대응 + 같은 가설 + 모든 작성 지지 근거 | 기존 수치 적용 | `검증 완료` |
| 정답 대응 + 같은 가설 + 지지 근거 일부 | 기존 수치 적용 | `대응 확인·근거 일부` |
| 정답 대응 + 같은 가설 + 지지 근거 없음 | 기존 수치 적용 | `대응 확인·근거 부족` |
| 정답 대응 + 다른 가설 | 기존 수치 적용 | `대응 확인·가설 불일치` |
| 선택 근거가 가설을 반증 | 대응 결과에 따름 | `선택 근거가 가설을 반증` |
| 오답 대응 | 기존 피해·학습 적용 | `현장 대응 실패` |

이 분리로 기존 플레이 기능을 약화하지 않으면서, 불충분한 추론이 공식 매뉴얼처럼 확정되는 것을 막는다.

### 영속 기록 계약

| 저장 항목 | 키 | 의미 | 갱신 규칙 |
|---|---|---|---|
| 공식 규칙 | `verified_rules[pattern_id]` | 가설·모든 작성 지지 근거·정답 대응이 함께 검증된 지식 | 같은 패턴의 검증 완료 시 최신 내용과 시도 횟수 갱신 |
| 검증 대기 후보 | `candidate_rules[pattern_id]` | 대응은 맞았지만 가설·근거가 불충분한 기록 | 공식 규칙이 없을 때만 최신 후보 유지 |
| 위험 사례 | `danger_cases[]` | 잘못된 대응과 그때 선택한 가설·근거 | 복합 ID 중복 제거, 반복 시 `attempts` 증가 |

`AfterlifeManualCatalog`는 조사 중 보여 주는 기존 사례·참고 자료로 유지한다. 플레이어가 이번 사건에서 만든 지식은 `anomaly_manual_records`가 단독 책임을 가지며 결과 화면·보고서·기록국 DB가 이 저장 데이터를 읽는다.

## 7. 로더 안전 경계

파일: `scripts/data/episode_loader.gd`

1. 기본 사건 JSON을 먼저 읽는다.
2. 일반 `.json` 사건에서만 같은 경로의 `_core_validation.json`을 확인한다.
3. 오버레이가 없거나 파싱되지 않으면 원본을 반환한다.
4. `target_episode_id`가 원본 ID와 다르면 오류를 남기고 원본을 반환한다.
5. `overrides`가 Dictionary가 아니면 원본을 반환한다.
6. 원본을 깊은 복사한 뒤 명시된 최상위 키만 대체한다.
7. 오버레이 자체를 직접 읽을 때 재귀 적용하지 않는다.
8. Godot 4.7.1의 경고-오류 승격 설정에서도 Variant 추론 경고가 발생하지 않도록 타입을 명시한다.

## 8. 적대적 검토와 조치

| 공격 | 발견 | 조치 | 잔여 위험 |
|---|---|---|---|
| 자동 예측이 정답을 대신함 | 패턴 설명이 행동을 암시할 수 있음 | 중립 질문으로 수정, 상세 예측은 매뉴얼 서랍으로 이동 | 문구 난이도는 플레이테스트 필요 |
| 가설 툴팁이 반증을 누설함 | `summary`에 `반증 후보`가 포함됨 | 선택 전 툴팁을 중립 고정 문구로 교체 | 없음 |
| 대응 툴팁이 추론을 누설함 | `reasoning`이 정답 이유를 설명함 | 선택 전 노출 금지, 결과 기록에서만 사용 | 없음 |
| UI가 마지막 매핑을 대신함 | `가설과 일치/불일치` 표식 | 표식 제거, 능력 종류만 표시 | 플레이어가 행동 문구를 실제로 읽는지 관찰 필요 |
| 근거 하나로 공식 검증됨 | 일부 지지만으로 `검증 완료` 가능 | 모든 작성 지지 근거를 선택해야 완료 | 없음 |
| 다른 사건이 새 필드를 요구함 | 비대상 사건에는 가설 데이터 없음 | 데이터 존재 여부로 직접 대응 폴백 | 새 사건 작성 가이드 필요 |
| 저장 중간 상태 증가 | 단계 상태 저장 유혹 | UI 파생 상태로 제한, Schema 유지 | 중간 단계 저장 요구가 생기면 별도 설계 필요 |
| 신규 보충 요원이 구형 테스트를 깨뜨림 | 과거 3인 정확표를 신규 요원에도 인덱싱 | 원래 3인은 정확값, 신규 요원은 공통 범위 검증 | 신규 고정표가 필요하면 별도 승인 |
| 공식 규칙이 재시도로 강등됨 | 승격 뒤 근거 일부 재시도가 후보를 다시 만들 수 있음 | 공식 규칙이 존재하면 후보를 재생성하지 않음 | 없음 |
| 가설과 대응의 근거가 뒤섞임 | 다른 가설을 고른 뒤 대응하면 실행 대응의 근거가 저장될 수 있음 | 저장 근거·추론은 선택 가설, 실행 결과는 대응으로 분리 | 없음 |
| 위험 사례가 반복 행으로 폭증 | 동일 오대응을 반복할 때 중복 기록 가능 | 복합 ID로 병합하고 `attempts`만 증가 | 위험 사례 정렬·필터는 후속 UX 범위 |
