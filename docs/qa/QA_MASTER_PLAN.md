> 문서 위치: `docs/qa/QA_MASTER_PLAN.md` | 상태 기준: `codex/mvp048-campaign-novel`의 `4cf39a2` | 상세 테스트 부록: [`TEST_CHECKLIST.md`](../../TEST_CHECKLIST.md)

## 1. QA 목표

QA는 버그를 나열하는 단계가 아니라 플레이어가 **읽고, 비교하고, 선택하고, 결과를 기록하는 흐름**을 실제로 완주할 수 있는지 증명한다. 구현 확인, 계획, 미검증을 분리하며 미실행 항목을 통과로 표시하지 않는다.

## 2. 공통 테스트 환경

| 항목 | 기준 |
|---|---|
| 자동 테스트 | 별도 임시 `APPDATA`에서 순차 실행; 사용자 저장을 읽거나 덮어쓰지 않음 |
| 엔진 확인 | Godot 4.7 headless 프로젝트 로드 |
| 해상도 | 1280×720, 1920×1080; 필요 시 사용자 환경 1918×943 |
| 입력 | 마우스, 키보드 포커스, Enter/Esc, 닫기·복귀 |
| 저장 | 신규/구형 저장 왕복, 재진입·재로드 중복 방지, 강제 종료 후 경계 확인 |
| 증거 | 테스트 로그, 관련 화면 캡처, 실패 재현 단계, 기준 SHA |

`docs/qa/MVP048_CAMPAIGN_ECONOMY_VALIDATION.md`와 기존 MVP QA 문서는 상세 실행 기록 부록이다. 이 표는 그 결과를 현재 작업자가 빠르게 판단하기 위한 요약이며, 새 결과가 나오면 사실만 갱신한다.

## 5. 수동 시각 QA 절차

각 대상 장면에서 1280×720과 1920×1080을 캡처한다.

1. 첫 화면에서 제목·목표·첫 선택·닫기/복귀가 보이는지 확인한다.
2. 한국어 3~6줄 지문, 긴 선택지, 비활성/위험 근거가 겹치거나 잘리지 않는지 확인한다.
3. 마우스와 키보드로 같은 진행이 가능한지, 팝업이 포커스를 가져가고 닫힌 뒤 원래 선택으로 돌아오는지 확인한다.
4. 매뉴얼/상세 정보가 선택지를 가리지 않는지, 저승역 전용 3열과 4×4 3열이 안전 프레임 안에 있는지 확인한다.
5. 캡처를 [이미지 인덱스](../art/IMAGE_INDEX.md)에 연결하고, 오래된 참고 이미지를 현행 증거로 오인하지 않게 상태를 표시한다.

## 6. 결함 보고와 완료 판정

결함은 `재현 단계 / 기대 / 실제 / 기준 SHA / 저장 상태 / 해상도·입력 / 캡처·로그 / 심각도`를 남긴다. 저장 손실, 진행 교착, 결과·비용 중복, 선택지 가림은 P0이다. 오탈자·비핵심 정렬은 플레이를 막는지에 따라 P1/P2로 분류한다.

목적 단위는 관련 자동 테스트, 저장 위험 테스트, `git diff --check`, 영향 화면 수동 QA를 통과해야 완료다. 확인하지 못한 것은 문서에 `미검증`으로 남기며, 다음 작업의 선행 조건으로 올린다.

## 7. 작업 때 함께 갱신할 것

규칙·UI·자산·사운드·저장이 바뀌는 모든 작업은 이 QA 기획서의 회귀 매트릭스와 현재 검증 표를 심사한다. 실행한 테스트와 미실행 수동 QA를 같은 커밋/PR 설명에 적는다. 문서 갱신의 전체 계약은 [프로그래밍 기획서](../programming/PROGRAMMING_ROADMAP_MVP.md#7-문서-동기화-계약)를 따른다.
